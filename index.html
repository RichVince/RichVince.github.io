Lab Title Examining Telnet and SSH in Wireshark Goal To demonstrate the security differences between Telnet and SSH by capturing and analyzing both types of remote sessions in Wireshark. The lab teaches why SSH is preferred for secure remote administration. Tools Used Security Workstation VM (analyst / cyberops) Wireshark (packet capture and protocol analysis) Telnet client SSH client What I Did Started Wireshark and captured traffic on the Loopback (lo) interface. Opened a Telnet session to localhost and logged in. Stopped the capture and used Follow → TCP Stream to view Telnet traffic. Observed that the entire Telnet session—including username and password—appeared in plaintext. Started a new Wireshark capture. Opened an SSH session to localhost, accepted the host key, and logged in. Stopped the capture and again used Follow → TCP Stream. Observed that SSH traffic was encrypted and unreadable. What I Observed Telnet Wireshark displayed all session data in plaintext, including: Username (with duplicated characters due to Telnet echo) Password (“cyberops” shown clearly) Terminal output The TCP stream was fully readable, confirming Telnet does not encrypt data. SSH Wireshark showed: SSH version exchange Key exchange algorithms Encrypted payloads The TCP stream was completely unreadable, showing only ciphertext. No credentials or commands were visible. Why It Matters Telnet sends all data unencrypted, making it vulnerable to: Credential theft Session hijacking Packet sniffing SSH encrypts: Usernames Passwords Commands Session output In a SOC or cybersecurity role, understanding this difference is critical because: Analysts must identify insecure protocols on networks. Attackers often exploit plaintext protocols like Telnet. Secure remote administration is essential for protecting systems. SSH is the industry standard because it prevents attackers from reading or manipulating remote sessions.
